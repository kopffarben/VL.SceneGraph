# Pipeline und Runtime

Dieses Dokument konsolidiert die Runtime- und Performance-relevanten Abschnitte der Architektur.
Es beschreibt den Weg vom immutablen Baum zur performanten Echtzeit-Verarbeitung bei 60fps.

---

## Flache BFS-Repräsentation (Performance-Kern)

### SoA-Layout (Struct of Arrays)

Separate Arrays pro Eigenschaft statt einem Array of Structs. Vorteile: Cache-Lokalität (nur benötigte Daten werden geladen), Prefetcher-Freundlichkeit, kein Padding-Overhead.

```csharp
public sealed class FlatSceneGraph : IDisposable
{
    public int NodeCount;
    public int Capacity;

    // Structural arrays (change on tree rebuild)
    public int[] ParentIndex;         // parent[i] < i (BFS invariant)
    public int[] FirstChildIndex;
    public int[] ChildCount;
    public int[] Depth;               // depth in tree (Root = 0)
    public NodeHandle[] Handles;      // mapping to logical node

    // Status arrays
    public byte[] DirtyFlags;         // bitfield per node
    public byte[] Visibility;

    // Level-based metadata
    public int[] LevelOffsets;        // LevelOffsets[d] = first index at depth d
    public int LevelCount;

    // Stream routing
    public StreamSlot[] PrimaryInputSlot;
    public StreamSlot[] PrimaryOutputSlot;
    public int[] CompositionInputStart;
    public int[] CompositionInputCount;
    public StreamSlot[] CompositionInputSlots;
    public float[] CompositionInputWeights;
    public BlendMode[] CompositionInputBlendModes;

    // Additional SoA arrays generated by [SceneComponent(FlatStorage = true)]
}
```

### BFS-Ordnung

Breitensuche (Level für Level): Root -> Kinder -> Enkel -> ...

Zentrales Invariant: **`ParentIndex[i] < i`** — der Parent steht immer vor dem Kind im Array.

Konsequenz: Eine einzige `for`-Schleife von vorne nach hinten (Top-Down) garantiert, dass der Parent-Zustand schon berechnet ist. Rückwärts (Bottom-Up) garantiert, dass alle Kinder schon verarbeitet sind.

### Level-basierte Parallelisierung

`LevelOffsets` ermöglicht Parallelisierung: Alle Nodes auf demselben Level können parallel verarbeitet werden (Abhängigkeiten bestehen nur zu Parents auf dem vorherigen Level). Threshold: Ab 256 Nodes pro Level lohnt sich `Parallel.For`.

### Inkrementeller Update

Drei Strategien je nach Art der Änderung:

1. **Property-Diff**: `ReferenceEquals` erkennt, ob sich die Struktur geändert hat. Wenn nur Properties geändert wurden: In-Place Update in den flachen Arrays.
2. **Subtree-Swap**: Wenn nur ein Teilbaum getauscht wird und die Größe gleich bleibt: In-Place kopieren.
3. **Full Rebuild**: Bei strukturellen Änderungen (Node add/remove): Komplett neu kompilieren.

## Multi-Pass Frame-Pipeline

### CompositorRuntime — Zwei Signaturen

Die Pipeline wird über zwei Methoden angesprochen. `Frame()` ist die interne Pipeline-Methode, `Update()` der VL-facing Process-Node-Wrapper:

```csharp
public sealed class CompositorRuntime : IDisposable
{
    // Internal pipeline execution — called per frame
    public (ReadOnlyMemory<object?> Outputs, ImmutableArray<SceneEdit> ClipEdits) Frame(
        SceneGraph graph, float globalTime, float deltaTime,
        InputContext inputContext, Frustum? frustum)

    // VL Process Node wrapper — calls Frame() internally
    public void Update(
        SceneGraph scene, float time, bool enabled,
        Frustum? frustum, InputContext inputContext,
        out ReadOnlyMemory<object?> outputs,
        out ImmutableArray<SceneEdit> clipEdits,
        out int nodeCount, out string frameStats)
}
```

`Frame()` liefert Outputs und ClipEdits als Tuple zurück. `Update()` entpackt diese in VL-kompatible `out`-Parameter und fügt Convenience-Werte wie `nodeCount` und `frameStats` hinzu.

### Pass-Reihenfolge (kritisch!)

Die 14 Passes laufen in strikter Reihenfolge auf dem Back-Buffer:

```csharp
public (ReadOnlyMemory<object?> Outputs, ImmutableArray<SceneEdit> ClipEdits) Frame(
    SceneGraph graph, float globalTime, float deltaTime,
    InputContext inputContext, Frustum? frustum)
{
    // Pass 1: Sync structure (compile tree -> flat arrays)
    SyncStructure(graph);

    // Pass 2: Time propagation (Top-Down)
    TimePass.PropagateTime(_back, globalTime, deltaTime);

    // Pass 3: External inputs (OSC, MIDI, Audio, sensors)
    ExternalInputPass.Evaluate(_back, inputContext);

    // Pass 4: Recording playback
    RecordingPass.Playback(_back, _recordingLibrary, _handleIndex);

    // Pass 5: State machines
    StateMachinePass.Evaluate(_back, deltaTime, inputContext);

    // Pass 6: Activity (fade-in/out, ControlMode dispatch)
    ActivityPass.UpdateActivity(_back, globalTime);

    // Pass 7: Timeline keyframes -> clip parameters
    TimelinePass.EvaluateKeyframes(_back);

    // Pass 8: Constraints (topologically sorted)
    ConstraintSolver.Resolve(_back);

    // Pass 9: Stream routing
    _streamPool.Cleanup();
    StreamRoutingPass.ComputeRouting(_back, _streamPool, _edgeIndex, _handleIndex);

    // Pass 10: Evaluation order + clip evaluation
    _clipEvaluator.ComputeEvaluationOrder(_back);
    SyncClipInstances(graph);
    _clipEvaluator.Evaluate(_back, deltaTime);

    // Pass 11: Recording capture
    RecordingPass.Record(_back, _activeRecorders, globalTime);

    // Pass 12: Optional 3D passes
    if (frustum.HasValue)
    {
        TransformPass.Execute(_back);
        BoundsPass.Execute(_back);
        VisibilityPass.Execute(_back, frustum.Value);
    }

    // Pass 13: Activity logging (for timeline UI)
    ActivityLogger.Record(_back, globalTime);

    // Pass 14: Dirty clear + buffer swap
    FlatPasses.ClearDirtyFlags(_back);
    (_front, _back) = (_back, _front);
    _streamPool.SwapBuffers();
}
```

### Top-Down Pass (Vorwärts-Iteration)

Für Propagation von Parent zu Kind (Transform, Zeit, Visibility):

```csharp
// Transform propagation, time propagation, visibility culling
for (int i = 1; i < nodeCount; i++)
{
    int p = parentIndex[i];
    // parentIndex[p] < i guaranteed: parent already computed
    worldTransform[i] = localTransform[i] * worldTransform[p];
}
```

### Bottom-Up Pass (Rückwärts-Iteration)

Für Aggregation von Kindern zum Parent (Bounding Boxes, Buffer-Füllung):

```csharp
// Bounding box aggregation, buffer filling
for (int i = nodeCount - 1; i >= 1; i--)
{
    int p = parentIndex[i];
    bounds[p] = BoundingBox.Merge(bounds[p], bounds[i]);
}
```

### Dirty-Propagation

Dirty-Flags propagieren Top-Down: Wenn ein Parent dirty ist, werden alle Kinder auch dirty. In einem typischen Frame ändern sich ca. 5% der Nodes — 95% der Arbeit wird übersprungen.

## Stream-Routing und Datenfluss

### StreamPool

Verwaltet typisierte Daten-Slots, die zwischen Nodes fließen. Referenz-gezählt für automatische Freigabe:

```csharp
public readonly record struct StreamSlot(int Index);

public sealed class StreamPool
{
    private SlotType[] _types;
    private object?[] _data;
    private int[] _refCount;

    public StreamSlot Allocate(SlotType type);
    public void Write(StreamSlot slot, object? data);
    public T? Read<T>(StreamSlot slot) where T : class;
    public void AddRef(StreamSlot slot);
    public void Release(StreamSlot slot);
    public void Cleanup(); // release unreferenced slots
}
```

### Routing-Logik

Die Routing-Bestimmung erfolgt in drei Schritten:

1. **Output-Slots allozieren** für alle aktiven Nodes.
2. **Input-Routing** pro Node bestimmen:
   - Explizite Edge vorhanden? -> Hat Vorrang.
   - Sonst: Hierarchie-basiertes Routing:
     - **Sequential**: Input vom vorherigen aktiven Sibling (oder Parent).
     - **Parallel**: Kein Input von Siblings, ggf. Parent-Input.
3. **Composition-Inputs** für Parents sammeln (Output-Slots aller aktiven Kinder).

### Composition-Modi

**Parallel**: Alle Kinder rendern unabhängig. Der Parent mischt die Ergebnisse (Layer-Compositing mit BlendMode und Opacity pro Kind).

**Sequential**: Output von Kind N ist Input von Kind N+1 (Effekt-Kette). Das letzte Kind in der Kette liefert den Output des Parents.

### Evaluation-Order

Bottom-Up: Blätter zuerst, dann aufwärts. Bei Sequential: Kinder in Reihenfolge. Inaktive Nodes werden übersprungen.

### Solo / Mute / Bypass

- **Mute**: Kind-Output wird ignoriert (nicht in Composition).
- **Solo**: Nur dieses Kind wird in die Composition einbezogen.
- **Bypass**: In Sequential-Chain: Input wird direkt als Output durchgereicht.

### Feedback-Loop

Output eines Frames wird zum Input des nächsten Frames. Funktioniert mit Double-Buffering: Der Front-Buffer hält die Outputs vom letzten Frame.

## Double-Buffering und Concurrency

### Prinzip

Zwei `FlatSceneGraph`-Instanzen: Front-Buffer (Render-Thread liest) und Back-Buffer (Update-Thread schreibt). Atomarer Swap am Frame-Ende:

```csharp
// Update thread writes to back buffer
// Render thread reads from front buffer
// At frame end:
(_front, _back) = (_back, _front);
```

### Immutabler Graph als natürliches Double-Buffering

Weil der logische `SceneGraph` immutable ist, hält der Renderer eine Referenz, die unbegrenzt gültig bleibt — keine Locks, keine zerrissenen Reads.

### ReadOnlySpan / ReadOnlyMemory

```csharp
public ReadOnlySpan<Matrix4x4> GetWorldTransforms()
    => _front.WorldTransform.AsSpan(0, _front.NodeCount);
```

`ReadOnlySpan<T>` kommuniziert Immutability-Intent. `Memory<T>` nur wenn Async-Grenzen überschritten werden (z.B. bei `Update()`-Outputs an VL).

## Memory-Management

### Ziel: Zero Allocations im Hot-Path

Kein `new`, kein Boxing, keine Closure-Captures im normalen Frame-Pfad. Alle temporären Buffer werden aus Pools bezogen.

### ArrayPool und RentedArray

Alle SoA-Arrays kommen aus `ArrayPool<T>.Shared`. `RentedArray<T>` als RAII-Wrapper für sichere Rückgabe:

```csharp
public ref struct RentedArray<T>
{
    private readonly T[] _array;
    private readonly ArrayPool<T> _pool;
    public readonly int Length;
    public Span<T> Span => _array.AsSpan(0, Length);
    public void Dispose() => _pool.Return(_array);
}
```

### Drei Stufen für temporäre Buffer

| Stufe | Größe | Mechanismus |
|-------|-------|-------------|
| 1 | < 1 KB | `stackalloc` |
| 2 | 1 KB – 1 MB | `ArrayPool<T>.Shared` |
| 3 | Persistent | Buffer die über Frames leben |

### Closure-Vermeidung

Lambdas, die lokale Variablen capturen, erzeugen Heap-Objekte. Lösung: Struct-basierte Callbacks mit generischem Interface:

```csharp
public interface INodeProcessor<TState>
{
    void Process(int nodeIndex, FlatSceneGraph graph, ref TState state);
}

// No heap allocation — struct is inlined by JIT
public struct AudioProcessor : INodeProcessor<float> { /* ... */ }
```

### Span-basierte Passes

Passes nehmen `Span<T>` statt rohe Arrays — erzwingt Bounds-Checking im Debug, ermöglicht JIT-Optimierungen (Bounds-Check Elimination bei bekannter Länge).

### Memory Budget Tracking

Debug-Modus: Zählt Pool-Rentals und Returns pro Frame. Ziel: Delta = 0 (alles was gemietet wird, wird im selben Frame zurückgegeben).
